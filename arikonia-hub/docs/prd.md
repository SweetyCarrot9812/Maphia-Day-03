# Arikonia Hub PRD

## 1. 제품 개요

### 1.1 배경 및 목적

**제품명**: Arikonia (아리코니아)
**어원**: 아리 (beautiful) + 코이노니아 (κοινωνία: fellowship, sharing, community)

**배경**:
- 현재 개별적으로 운영 중인 자기계발 학습 프로젝트들(Care-Lit, Tem-Flow, Arisper)을 하나의 생태계로 통합
- 사용자가 여러 프로젝트를 이용할 때마다 중복 회원가입과 로그인을 해야 하는 불편함 해소
- 구독 기반 비즈니스 모델 구축을 통한 지속 가능한 서비스 운영

**핵심 문제**:
- 프로젝트별 분산된 인증 시스템
- 통합된 사용자 관리 및 구독 관리 부재
- 프로젝트 간 연결고리 부족
- 브랜드 정체성 부재

**해결 방안**:
- SSO(Single Sign-On)를 통한 통합 인증
- 중앙 집중식 구독 및 접근 권한 관리
- 프로젝트 포털 역할의 Hub 사이트 구축
- 통합 브랜드(Arikonia) 하의 일관된 사용자 경험

### 1.2 목표

**비즈니스 목표**:
- Phase 1: 개별 프로젝트 완성도 향상에 집중
- Phase 2: 구독 기반 수익 모델 구축
- Phase 3: 사용자 기반 확대 및 생태계 성장

**사용자 목표**:
- 하나의 계정으로 모든 Arikonia 프로젝트 접근
- 구독 플랜에 따른 유연한 프로젝트 이용
- 각 프로젝트에서 완전히 개인화된 학습 경험
- 깔끔하고 직관적인 프로젝트 탐색 및 접근

### 1.3 성공 지표

**Phase 1 (MVP)**:
- SSO 인증 성공률 > 99%
- 프로젝트 접근 전환율 > 80%
- 사용자 계정 생성 완료율 > 70%
- 다크모드 사용률 측정

**Phase 2 (Enhancement)**:
- 유료 구독 전환율 > 5%
- 구독 유지율 > 80%
- 프로젝트 간 교차 이용률 > 30%

**정성적 성공 기준**:
- "하나의 계정으로 편리하다"는 사용자 피드백
- 프로젝트 간 일관된 브랜드 경험 인식
- "신뢰할 수 있는 교육 플랫폼"이라는 평가

### 1.4 범위

**Phase 1 MVP - 포함 사항 (In Scope)**:
- Arikonia Hub 랜딩 페이지 (arikonia.com)
- SSO 통합 인증 시스템 (회원가입/로그인)
- 3개 프로젝트 소개 및 연결 (Care-Lit, Tem-Flow, Arisper)
- 구독 플랜 구조 (free/basic/premium/enterprise)
- 프로젝트별 접근 권한 관리 (RLS 기반)
- 다크모드 지원
- 사용자 대시보드 (구독 정보, 프로젝트 접근 현황)
- 관리자 계정 (tkandpf18@naver.com)

**Phase 1 MVP - 제외 사항 (Out of Scope)**:
- 결제 시스템 통합 (Toss Payments, Stripe 등)
- 구독 플랜 변경 기능
- 프로젝트 간 학습 데이터 공유
- 프로젝트 간 통합 통계
- 이메일 마케팅
- 모바일 앱
- 소셜 로그인 (Google, Kakao 등)

**Phase 2 - 추가 예정**:
- 결제 시스템 통합
- 구독 관리 (플랜 변경, 결제 내역, 영수증)
- 프로모션 코드 및 쿠폰
- 추가 프로젝트 (EconoVerse, VocalSphere, FaithBridge)

## 4. 기능 요구사항

### 4.1 주요 기능

#### F-001: Arikonia Hub 랜딩 페이지
- **설명**: 브랜드 소개, 프로젝트 카드, 비전 제시
- **우선순위**: P0 (필수)
- **예상 공수**: 완료됨
- **의존성**: 없음
- **상세**:
  - Hero 섹션 (Arikonia 브랜드 설명)
  - 프로젝트 카드 3개 (Care-Lit, Tem-Flow, Arisper)
  - 각 카드: 이름(한글/영문), 태그라인, 설명, 상태, 아이콘, URL
  - Vision 섹션 (아리코니아가 추구하는 가치)
  - 반응형 디자인

#### F-002: SSO 통합 인증
- **설명**: Supabase Auth 기반 회원가입/로그인
- **우선순위**: P0 (필수)
- **예상 공수**: 완료됨
- **의존성**: F-003 (데이터베이스 스키마)
- **상세**:
  - 이메일 + 비밀번호 회원가입
  - 이메일 인증
  - 로그인/로그아웃
  - 세션 관리 (Zustand)
  - 자동 프로필 생성 (trigger)

#### F-003: 데이터베이스 스키마
- **설명**: 사용자, 구독, 프로젝트, 접근 권한 관리
- **우선순위**: P0 (필수)
- **예상 공수**: 완료됨 (실행 대기 중)
- **의존성**: 없음
- **상세**:
  - users 테이블
  - subscription_plans 테이블 (4개 플랜)
  - user_subscriptions 테이블
  - projects 테이블 (3개 프로젝트)
  - plan_project_access 테이블 (플랜별 접근 권한)
  - user_project_access 테이블 (사용자별 커스텀 권한)
  - RLS 정책
  - check_project_access() 함수

#### F-004: 다크모드
- **설명**: 라이트/다크 테마 전환
- **우선순위**: P0 (필수)
- **예상 공수**: 완료됨
- **의존성**: 없음
- **상세**:
  - CSS 변수 기반 테마
  - localStorage 저장
  - 토글 버튼 (Sun/Moon 아이콘)
  - 모든 페이지/컴포넌트 대응

#### F-005: 사용자 대시보드
- **설명**: 구독 정보, 접근 가능 프로젝트 확인
- **우선순위**: P0 (필수)
- **예상 공수**: 2일
- **의존성**: F-002, F-003
- **상세**:
  - 현재 구독 플랜 표시
  - 만료일 (있는 경우)
  - 접근 가능한 프로젝트 목록
  - 각 프로젝트로 바로가기 버튼
  - 프로필 정보 (이메일, 닉네임)

#### F-006: 프로젝트 접근 제어
- **설명**: 구독 플랜에 따른 프로젝트 접근 권한 확인
- **우선순위**: P0 (필수)
- **예상 공수**: 1일
- **의존성**: F-002, F-003
- **상세**:
  - check_project_access() RPC 호출
  - 접근 불가 시 업그레이드 안내
  - 접근 가능 시 해당 프로젝트 URL로 이동

#### F-007: 관리자 기능
- **설명**: 사용자 관리, 구독 관리
- **우선순위**: P1 (중요)
- **예상 공수**: 3일
- **의존성**: F-002, F-003
- **상세**:
  - 사용자 목록 조회
  - 사용자 구독 플랜 변경
  - 사용자별 커스텀 접근 권한 부여
  - 프로젝트 활성화/비활성화

## 5. 사용자 여정 (User Journey)

### Segment 1: 신규 사용자 (Care-Lit 관심)

#### 시나리오: 간호학 학습을 위해 Care-Lit을 처음 발견한 학생

**여정 단계:**

1. **발견** - [페이지: Arikonia 랜딩 (arikonia.com)]
   - 사용자 행동:
     - Google 검색 "간호사 국가고시 학습 플랫폼" → Arikonia 발견
     - 랜딩 페이지 방문
     - 프로젝트 카드 탐색
   - 시스템 반응:
     - 브랜드 스토리 제시 (아리코니아 의미)
     - Care-Lit 카드 강조 ("운영 중" 배지)
     - Vision 섹션으로 신뢰감 형성
   - 감정 상태: 호기심, 기대감
   - 터치포인트: Hero 섹션, 프로젝트 카드

2. **탐색** - [페이지: Care-Lit 상세 or 랜딩 페이지]
   - 사용자 행동:
     - Care-Lit 카드 클릭
     - 무료 체험 가능 여부 확인
   - 시스템 반응:
     - "무료 플랜으로 시작하기" 안내
     - 회원가입 유도
   - 감정 상태: 신중함, 확신 필요
   - 터치포인트: CTA 버튼, 무료 플랜 안내

3. **가입** - [페이지: 회원가입 (/signup)]
   - 사용자 행동:
     - 이메일, 닉네임, 비밀번호 입력
     - 가입 완료
   - 시스템 반응:
     - 자동으로 free 플랜 할당
     - 이메일 인증 메시지 표시
     - 대시보드로 리다이렉트
   - 감정 상태: 기대감, 약간의 불안 (이메일 인증)
   - 터치포인트: 회원가입 폼, 성공 메시지

4. **첫 접근** - [페이지: 대시보드 (/dashboard)]
   - 사용자 행동:
     - 구독 정보 확인
     - "Care-Lit 시작하기" 버튼 클릭
   - 시스템 반응:
     - 현재 플랜: 무료 (제한적 접근)
     - carelit.arikonia.com으로 이동
     - SSO 토큰 전달
   - 감정 상태: 설렘, 몰입 준비
   - 터치포인트: 대시보드 프로젝트 카드

5. **학습 시작** - [페이지: Care-Lit 앱]
   - 사용자 행동:
     - Care-Lit에서 자동 로그인 확인
     - 20문제 제한 내 학습
   - 시스템 반응:
     - "무료 플랜: 20문제까지" 안내
     - 학습 진행
   - 감정 상태: 만족, 더 많은 기능 원함
   - 터치포인트: Care-Lit 앱 내 제한 안내

6. **업그레이드 고려** - [페이지: Care-Lit 또는 대시보드]
   - 사용자 행동:
     - 20문제 완료 후 제한 메시지
     - "플랜 업그레이드" 버튼 클릭
   - 시스템 반응:
     - Phase 1: 업그레이드 안내 페이지 (결제 없음)
     - Phase 2: 결제 페이지
   - 감정 상태: 가치 인정, 결제 의향
   - 터치포인트: 업그레이드 CTA

#### 예상 Pain Points
- 이메일 인증 과정에서 이탈 가능성
- 무료 플랜 제한이 너무 빨리 도달하면 불만
- Care-Lit으로 이동 시 새 탭에서 열리는 것에 대한 혼란

#### 최적화 포인트
- 회원가입 시 "인증 메일 전송됨, 스팸함 확인" 명확한 안내
- 무료 플랜에서 학습 진행률과 남은 문제 수 실시간 표시
- Care-Lit 이동 시 "새 창에서 열립니다" 툴팁

   - 터치포인트: 로그인 상태 헤더

2. **프로젝트 발견** - [페이지: 랜딩 or 대시보드]

   - 감정 상태: 호기심, 제약 인지
   - 터치포인트: Tem-Flow 카드

3. **접근 시도** - [페이지: 대시보드 or 접근 제한 모달]
   - 사용자 행동:
     - "Tem-Flow 시작하기" 클릭
   - 시스템 반응:
     - check_project_access() 확인
     - 접근 불가 시: "Premium 플랜이 필요합니다" 모달
     - 접근 가능 시: temflow.arikonia.com 이동
   - 감정 상태: 좌절 or 기대
   - 터치포인트: 접근 제한 안내

4. **플랜 업그레이드** - [페이지: Phase 2 결제 페이지]
   - 사용자 행동:
     - "Premium으로 업그레이드" 버튼 클릭
     - (Phase 2에서) 결제 진행
   - 시스템 반응:
     - Phase 1: "곧 출시됩니다" 안내
     - Phase 2: 결제 처리 후 구독 업데이트
   - 감정 상태: 가치 판단 중
   - 터치포인트: 플랜 비교 페이지

#### 예상 Pain Points

- 여러 프로젝트 이용 시 가격 부담

#### 최적화 포인트
- 플랜 비교표 명확하게 제시

## 6. Information Architecture (IA)

```
[Arikonia Hub - arikonia.com]
├── 홈 (/)
│   ├── Hero 섹션
│   ├── 프로젝트 카드 (3개)
│   │   ├── Care-Lit → carelit.arikonia.com
│   │   ├── Tem-Flow → temflow.arikonia.com
│   │   └── Arisper → arisper.arikonia.com
│   ├── Vision 섹션
│   └── Footer
├── 인증 (/login, /signup)
│   ├── 로그인 폼
│   ├── 회원가입 폼
│   └── 비밀번호 재설정 (Phase 2)
├── 대시보드 (/dashboard)
│   ├── 구독 정보
│   ├── 프로필
│   ├── 접근 가능 프로젝트
│   └── 빠른 시작 버튼
└── 관리자 (/admin)
    ├── 사용자 관리
    ├── 구독 관리
    └── 프로젝트 관리

[Care-Lit - carelit.arikonia.com]
├── 독립적인 앱 구조
├── SSO 토큰으로 자동 로그인
└── 학습 데이터는 Care-Lit DB에 저장

[Tem-Flow - temflow.arikonia.com]
└── (동일 구조)

[Arisper - arisper.arikonia.com]
└── (동일 구조)
```

**네비게이션 구조:**
- **Primary Navigation**:
  - Arikonia Hub: 홈, 로그인, 대시보드
  - 각 프로젝트: 독립적인 네비게이션
- **Secondary Navigation**:
  - 사용자 프로필 드롭다운 (로그아웃, 설정)
  - 다크모드 토글
- **Deep Links**:
  - 각 프로젝트 카드에서 직접 외부 URL로 연결

**페이지 계층:**
- **Level 1 (Entry)**: 랜딩 페이지 (/)
- **Level 2 (Auth)**: 로그인, 회원가입
- **Level 3 (User Area)**: 대시보드, 프로필
- **Level 4 (Admin)**: 관리자 패널
- **External**: 각 프로젝트 (별도 도메인)

## 7. 기술 요구사항

### 7.1 기술 스택

**Frontend**:
- Next.js 15.5.6 (App Router, RSC)
- React 19.1.0
- TypeScript 5.x
- Tailwind CSS 4.0 (@tailwindcss/postcss)
- shadcn/ui (Radix UI 기반)
- Zustand 5.0.8 (상태 관리)

**Backend**:
- Next.js API Routes (서버리스)
- Supabase (PostgreSQL + Auth + Storage)

**Database**:
- PostgreSQL (Supabase)
- Row Level Security (RLS)
- JSONB for flexible features

**Infrastructure**:
- Vercel (호스팅, CDN)
- Supabase Cloud (데이터베이스, 인증)
- hosting.kr (도메인 DNS)

### 7.2 통합/연동

**외부 서비스**:
- Supabase Auth (이메일/비밀번호 인증)
- Supabase Database (PostgreSQL)
- (Phase 2) Toss Payments or Stripe (결제)

**내부 시스템**:
- Care-Lit: SSO 토큰 기반 자동 로그인
- Tem-Flow: SSO 토큰 기반 자동 로그인
- Arisper: SSO 토큰 기반 자동 로그인
- 각 프로젝트는 check_project_access() 호출로 접근 권한 확인

### 7.3 성능 요구사항

**로딩 시간**:
- 랜딩 페이지 First Contentful Paint < 1.5초
- 대시보드 로드 < 2초
- 프로젝트 이동 < 1초 (SSO 토큰 전달)

**동시 사용자**:
- Phase 1: ~100명 (베타)
- Phase 2: ~1,000명
- Phase 3: ~10,000명

**데이터 처리**:
- 사용자 인증: < 500ms
- 접근 권한 확인: < 200ms
- 대시보드 데이터 로드: < 1초

### 7.4 보안 요구사항

**인증/인가**:
- Supabase Auth with JWT
- httpOnly 쿠키 (선택적)
- Row Level Security (RLS) 모든 테이블

**데이터 보호**:
- HTTPS only (Vercel 자동)
- 비밀번호 bcrypt 해싱 (Supabase 자동)
- SQL Injection 방지 (Parameterized queries)
- XSS 방지 (React 자동 escaping)

**컴플라이언스**:
- (Phase 2) 개인정보처리방침
- (Phase 2) 이용약관
- GDPR 고려 (유럽 사용자 시)

## 8. 제약사항 및 가정

### 제약사항
- Phase 1에서는 결제 시스템 없음 (수동 플랜 업그레이드)
- 각 프로젝트는 별도 서브도메인으로 운영 (기술 스택 독립)
- Care-Lit, Tem-Flow, Arisper는 개발 중 (완성도 우선)
- 모바일 앱 없음 (반응형 웹만)
- 소셜 로그인 없음 (이메일/비밀번호만)

### 가정
- 사용자는 이메일 인증을 완료한다고 가정
- 각 프로젝트는 SSO 토큰을 올바르게 검증한다고 가정
- 사용자는 브라우저 쿠키를 허용한다고 가정
- Supabase와 Vercel은 99.9% 가용성을 제공한다고 가정
- 사용자는 한국어 사용자 중심 (영어 지원은 Phase 3)

## 9. 출시 계획

### Phase 1: MVP (현재)

**포함 기능**:
- Arikonia Hub 랜딩 페이지
- SSO 통합 인증 (회원가입/로그인)
- 구독 플랜 구조 (수동 관리)
- 프로젝트 접근 권한 관리
- 사용자 대시보드
- 다크모드
- 관리자 기능 (수동 SQL)

**일정**: 2주
- Week 1: 데이터베이스 스키마 실행, 대시보드 구현
- Week 2: 관리자 페이지, 접근 제어 테스트, 배포

**목표**:
- 사용자가 하나의 계정으로 모든 프로젝트 접근 가능
- 구독 플랜에 따른 접근 제어 작동
- 안정적인 SSO 경험

### Phase 2: Enhancement (3-6개월 후)

**추가 기능**:
- 결제 시스템 통합 (Toss Payments)
- 구독 관리 페이지 (플랜 변경, 결제 내역)
- 프로모션 코드 시스템
- 이메일 알림 (구독 만료, 결제 실패)
- 관리자 대시보드 (UI 기반)
- 사용자 피드백 시스템

**일정**: 4주

**목표**:
- 자동화된 결제 및 구독 관리
- 사용자 이탈률 < 20%
- 유료 전환율 > 5%

### Phase 3: Scale (6-12개월 후)

**최적화/확장**:
- 추가 프로젝트 3개 출시 (EconoVerse, VocalSphere, FaithBridge)
- 소셜 로그인 (Google, Kakao)
- 영어 다국어 지원
- 모바일 앱 (React Native or Flutter)
- 프로젝트 간 크로스 프로모션
- 사용자 추천 시스템

**일정**: 12주

**목표**:
- 사용자 기반 10,000명
- 프로젝트 간 교차 이용률 > 30%
- 글로벌 사용자 > 10%

## 10. 위험 요소 및 완화 방안

| 위험 요소 | 발생 가능성 | 영향도 | 완화 방안 |
|-----------|-------------|--------|----------|
| 이메일 인증 이탈률 높음 | 높음 | 중간 | 명확한 안내 메시지, 스팸함 확인 알림 |
| Supabase 장애 | 낮음 | 높음 | 백업 계획, 상태 페이지 모니터링 |
| SSO 토큰 검증 실패 | 중간 | 높음 | 각 프로젝트별 토큰 검증 테스트 철저히 |
| 프로젝트 미완성으로 사용자 불만 | 중간 | 중간 | "베타" 배지 명시, 기대치 관리 |
| 무료 플랜 남용 | 낮음 | 낮음 | Rate limiting, abuse 모니터링 |
| RLS 정책 누락으로 데이터 노출 | 낮음 | 높음 | 철저한 RLS 정책 테스트, 보안 감사 |
| 다크모드 일부 페이지 미적용 | 중간 | 낮음 | 모든 페이지 dark: 클래스 체크리스트 |
